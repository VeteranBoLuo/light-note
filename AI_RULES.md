# Light Note AI 开发规则

本文件用于约束 AI 在 `light-note` 项目中的修改方式。除非用户明确说明可以忽略，否则默认必须遵守。

## 1. 多端优先

- 项目同时包含 PC 端和移动端。
- 新增功能、修改交互、调整布局时，默认必须同时考虑 PC 端和移动端，不能只改一端。
- 如果需求没有特别说明“只改移动端”或“只改 PC 端”，则两端都要检查。
- 本项目当前响应式分档统一以视口宽度为准，不按 UA 判断设备。
- 统一断点来源：`src/store/bookmark.ts`
  - `width < 768`：`mobile`
  - `768 <= width < 1200`：`tablet`
  - `width >= 1200`：`desktop`
- 做适配时，优先复用 `bookmarkStore` 中已有 getter：
  - `bookmark.isMobile`
  - `bookmark.isTablet`
  - `bookmark.isDesktop`
  - `bookmark.isMobileDevice`
- 不要在新代码里重复发明新的断点标准，也不要再次使用 UA 检测去区分手机、平板、桌面。

## 2. 主题规则

- 项目主题切换基于 `data-theme`。
- 主题实际挂载位置：`document.documentElement.setAttribute('data-theme', user.currentTheme)`。
- 主题应用逻辑位置：`src/App.vue`
- 主题变量配置文件：`src/assets/css/theme.less`
- 开发时必须优先使用主题变量，不要直接写死一套浅色/深色样式。
- 允许使用的方式：
  - 使用 `var(--xxx)` 主题变量
  - 在 `theme.less` 中补充新的主题变量，再在组件内消费
- 不推荐的方式：
  - 在业务组件里硬编码一大段白天/夜间颜色
  - 通过 JS 在组件内手动判断 `day/night` 后拼接大量行内样式
- 主题状态来源统一使用用户 store，不要自行维护第二套主题状态。
- 主题相关 store：`src/store/useUser.ts`

## 3. 国际化规则

- 项目使用 `vue-i18n` 做国际化。
- i18n 目录：`src/i18n`
- 入口文件：`src/i18n/index.ts`
- 语言文件：
  - `src/i18n/locales/zh-CN.ts`
  - `src/i18n/locales/en-US.ts`
- 每次新增页面、修改页面、增加弹窗、按钮、标题、提示语、空状态、表单文案时，都必须考虑国际化。
- 只要是页面上会显示给用户的固定文案，默认都不能直接写死在模板里。
- 修改文案时，必须同步补齐中英文两个语言文件，避免只改中文不改英文。
- 如果页面里出现暂时无法国际化的服务端返回文本，需要在规则允许范围内尽量包裹或转换，不要继续扩大硬编码文案范围。

## 4. 路由与多端页面映射

- 本项目存在桌面端路由和移动端路由映射逻辑。
- 相关位置：`src/App.vue`
- 重点映射对象：
  - `phoneReplaceMap`
  - `deskReplaceMap`
- 如果新增的是带明显端差异的页面、入口或跳转逻辑，需要检查是否要补充桌面端和移动端之间的路由映射。
- 如果新增页面只在某一端存在，也要确认切换断点时是否会出现错误跳转、空页面或死路由。

## 5. 状态与现有能力复用

- 项目使用 Vue 3 + TypeScript + Pinia + Vue Router + Ant Design Vue。
- 共享状态优先复用现有 store，不要随意新增重复状态。
- 常用 store 位置：
  - `src/store/bookmark.ts`
  - `src/store/useUser.ts`
  - `src/store/note.ts`
  - `src/store/cloudSpace.ts`
- 已存在公共组件、基础组件、图标配置时，优先复用，不要重复造轮子。
- 常见复用位置：
  - `src/components/base`
  - `src/config/icon.ts`

## 6. 样式修改原则

- 修改样式时，先确认是否已有对应主题变量、公共类、布局模式可复用。
- 涉及颜色、背景、边框、阴影时，优先走 `theme.less` 变量。
- 涉及响应式时，优先复用现有断点体系，不要再写一套新的“神秘尺寸”标准。
- 改布局时，不只看当前页面是否正常，还要注意是否会影响顶部导航、移动端抽屉、搜索框、个人中心、右上角操作区等已有结构。

## 7. 文案与交互

- 新增交互时，要保证 PC 端和移动端都能完成核心操作。
- hover 才能触发的能力，要额外考虑移动端是否有等价入口。
- 弹层、下拉、tooltip、popover 这类交互，要注意是否和现有浮层容器、z-index、导航栏区域发生遮挡冲突。

## 8. 修改后自检

- 只要改动了前端功能，默认至少检查以下内容：
  - PC 端是否正常
  - 移动端是否正常
  - 主题切换后是否正常
  - 中英文是否正常
- 如果改动涉及路由、导航、顶部栏、浮层、搜索、个人中心、笔记详情等高频区域，必须额外检查是否出现遮挡、错位、重叠、无法点击的问题。

## 9. 构建与环境

- 项目前端构建命令：`pnpm build`
- 推荐 Node 版本：`20.x`
- `package.json` 已声明 `node: 20.x`，开发和验证时应尽量遵循。

## 10. 默认执行原则

- 默认做法不是“只把功能做出来”，而是“按项目现有规则做完整”。
- 除非需求明确说明，否则 AI 每次改动都应同时考虑：
  - 多端适配
  - 主题兼容
  - 国际化
  - 现有路由与交互体系
- 如果某次需求确实只需要改单端、跳过国际化、忽略主题，必须由用户明确说明后才能省略。
